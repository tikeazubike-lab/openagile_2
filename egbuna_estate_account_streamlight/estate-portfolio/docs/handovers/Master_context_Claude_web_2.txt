## CI/CD

### Branch Flow

```text
feature/*, develop
    → static analysis
    → tests
    → build

test
    → static analysis
    → tests
    → build
    → deploy
    → e2e

main
    → static analysis
    → tests
    → build
    → approval
    → deploy
    → e2e
```

### Approval Mechanism (main only)

```text
Manual sign-off comment ("/approve") on the PR, posted by an authorized human,
verified by CI before the deploy job is permitted to run.
```

Not GitHub branch-protection required-reviewers. Not automatic. See HO-027 for implementation.

Rules:
- No direct merge to main
- Review required every cycle
- Deploy job on main MUST be gated on the `/approve` comment check — no other trigger authorizes it

## Handoff Rules

### DeepSeek Pro / Claude Web (Architects)

Never implement before design approval. Design approval requires **both** architects — one architect's approval does not authorize implementation.

### Hermes (OpenCode Zen)

Never implements. Reviews OpenCode Go builder output against: infra rules, locked architectural decisions, CI/CD contract. Blocks merge on violation. Signs off via structured `HO-*.md` artifact.

### OpenCode Go Builders (`deepseek-v4-flash`, others)

Never start backend work before:
- contracts/specs exist
- acceptance criteria exist

Prefer: Gherkin RED phase completed.

Never start frontend before APIs are stable.

### Lovable

Allowed: frontend generation
Forbidden: backend, infrastructure, databases, deployment

### Reviews
- Nothing merges directly to main
- Previous output never authorizes skipping review
- Main additionally requires the `/approve` comment gate — see CI/CD
