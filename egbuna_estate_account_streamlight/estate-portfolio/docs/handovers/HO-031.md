## 1. What Changed & Why

**Priority shift:** We deprioritized the ordered Phase 3C plan (NAV History → Transactions → Watchlist → Dividends → Claims) in favor of building all remaining pages **minimally first**, then cycling back for polish. The old plan was spec-led to production depth per feature — slow and blocking. The new approach: one page at a time, working data + UI only, then deploy. This matches the user's preferred workflow ("build all pages minimally first, then cycle back for details/bugs/drill-downs").

**Why start with Claims?** The user had a Lovable-downloaded React template for a Dividend Navigator dashboard (724 lines, mock data, chart-heavy). Porting it into EPM as the Claims page was the fastest route to a visually rich, data-driven page that demonstrates the model. Claims = Dividends in EPM's domain model (the ClaimRecord tracks unclaimed/claimed dividend payouts). So F-010 replaces both the old F-010 (Claims) and F-008 (Dividends) in one shot.

## 2. Architecture

| Component | Implementation | Notes |
|---|---|---|
| **Route** | `/_app/claims` → `_app.claims.tsx` | Sidebar nav replaces "Dividends" (old stub page `/dividends` still exists, hidden from nav) |
| **Backend** | `GET /api/v1/claims` — 4 queries: total stats, registrar stats, records list, registrars list | Uses chained eager loads: `ClaimRecord → Holding → Company → Registrar` to avoid N+1 |
| **Pydantic** | `ClaimResponse`, `EnvelopeResponse` schemas | Wraps response in `_envelope({data: ..., meta: ..., error: ...})` |
| **Status Filter** | Comma-separated `in_()` — e.g. `?status=Unclaimed,Claimed` | Client sends single or multi |
| **Search** | Client-side filter (TanStack Query + JS `filter()`), no pagination in v1 |
| **Charts** | 3 Recharts widgets: Bar chart (registrars × unclaimed), Pie chart (claimed vs unclaimed), KPI cards |
| **Registrar Stats** | Computed client-side from all claims records | Server-side aggregate endpoint deferred to v2 |
| **Drawers** | 2 Sheet drawers: Claim Detail (full record) + Registrar Profile (stats + related claims) |

## 3. Architect Review Highlights

The spec was submitted to DeepSeek Architect (DeepSeek Pro on OpenRouter). Key findings incorporated:

1. **Critical:** Chained eager loading for `ClaimRecord → Holding → Company → Registrar` (was lazy-loading, N+1 risk)
2. **Critical:** Pydantic response schema + `_envelope()` wrapping (was returning raw ORM objects)
3. **High:** Comma-separated status filter (`?status=Unclaimed,Claimed`)
4. **High:** Client-side search (no backend search endpoint — v1 limitation documented)
5. **Medium:** Loading/error/empty states, account # synthetic composite, mandate documented as v1 limitation
6. **Low:** Registrar stats computed client-side (server enrichment deferred), CSV export deferred to v2

## 5. Known Limitations (v1 Scope)

| Limitation | Reason | Plan |
|---|---|---|
| No pagination | Claims count is small (< 200) | Add when needed |
| No server-side search | Client-side filter suffices for v1 | Add if performance drops |
| No CSV export | Button present but non-functional | v2 feature |
| Registrar stats computed client-side | Avoids extra endpoint complexity | Move to backend when perf required |
| No automated tests | Acceptance test checklist created separately | Manual run + automated in CI |
| `Account #` is synthetic (company + holding_id) | No single account number in data model | Formalize when account model exists |
| E-Dividend Mandate not tracked in DB | No mandate field on ClaimRecord | v2 enhancement |

## 6. Theme Compatibility

All colors use EPM's CSS variable system:
- Backgrounds: `var(--bg-surface)`, `var(--bg-subtle)`, `var(--bg-canvas)`
- Text: `var(--text-primary)`, `var(--text-muted)`
- Accents: `var(--accent-lavender)`, `var(--accent-green)`, `var(--accent-amber)`
- Chart colors: `var(--chart-1)` through `var(--chart-5)` (mapped to EPM accent colors)
-Info/pending: `var(--color-info)` → `var(--accent-amber)`

Both light and dark themes are fully supported.
