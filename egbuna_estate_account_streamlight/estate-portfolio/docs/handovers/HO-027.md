## 1. Objective

Add a GitHub Actions workflow that runs static analysis, tests, and a build check on every PR targeting the `test` branch, posts results as a PR comment, and (optionally) logs results to the test builder API. This is the CI/CD contract's `test → static analysis → tests → build` stage, implemented as PR gate checks.

## 4. Implementation Steps

### 4.1 Register self-hosted runner on VPS (one-time, manual — not part of workflow YAML)

```bash
mkdir -p ~/actions-runner && cd ~/actions-runner
curl -o actions-runner-linux-x64.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.319.1/actions-runner-linux-x64-2.319.1.tar.gz
tar xzf actions-runner-linux-x64.tar.gz

./config.sh --url https://github.com/<org>/<repo> --token <REGISTRATION_TOKEN> \
  --labels vps,epm --name epm-vps-runner --unattended

sudo ./svc.sh install
sudo ./svc.sh start
sudo usermod -aG docker $(whoami)
```

Confirm the runner process user has `docker` group membership — verify with `docker ps` as that user before proceeding, not just after `usermod` (group changes require a new session).

### 4.2 Create test database (one-time, against EXISTING instance)

```bash
docker exec -it <existing-postgres-container> psql -U postgres <<'EOF'
CREATE DATABASE epm_test OWNER epm_app;
EOF
```

Confirm this creates a database *inside* the existing instance, not a new container — verify with `docker ps` showing no new Postgres container before/after.

### 4.3 Add `docker-compose.ci.yml`

Must define a `backend` service that:
- Shares the same base image/build context as the main `docker-compose.yml` backend service
- Points `DATABASE_URL` at `epm_test`, not the production database
- Attaches to `openagile_network` (reuse, do not redeclare if already external)

Flag in your HO response whether this required duplicating dependency lists from the main compose file, and if so, whether a shared base/override pattern (`docker-compose.yml` + `docker-compose.ci.override.yml`) would reduce drift risk going forward — this is a design question back to me if the duplication is significant, not something to resolve unilaterally.

### 4.4 Create `.github/workflows/test-pr.yml`

```yaml
name: PR Checks (test branch)

on:
  pull_request:
    branches: [test]

concurrency:
  group: pr-checks-${{ github.event.pull_request.number }}
  cancel-in-progress: true

jobs:
  static-analysis:
    runs-on: [self-hosted, epm]
    steps:
      - uses: actions/checkout@v4

      - name: Ruff (lint)
        run: |
          docker compose -f docker-compose.ci.yml run --rm backend \
            ruff check .

      - name: Mypy (type check)
        run: |
          docker compose -f docker-compose.ci.yml run --rm backend \
            mypy app/

  tests:
    needs: static-analysis
    runs-on: [self-hosted, epm]
    env:
      DATABASE_URL: postgresql://epm_app:${{ secrets.EPM_TEST_DB_PASSWORD }}@postgres:5432/epm_test
    steps:
      - uses: actions/checkout@v4

      - name: Ensure backend network exists
        run: docker network inspect openagile_network >/dev/null 2>&1 || \
             docker network create openagile_network

      - name: Reset test schema (truncate, never drop)
        run: |
          docker compose -f docker-compose.ci.yml run --rm backend \
            python -m app.scripts.reset_test_schema

      - name: Run migrations (additive-only)
        run: |
          docker compose -f docker-compose.ci.yml run --rm backend \
            alembic upgrade head

      - name: Run pytest with JSON report
        run: |
          docker compose -f docker-compose.ci.yml run --rm backend \
            pytest backend/tests/ \
              --json-report --json-report-file=/tmp/report.json \
              --junitxml=/tmp/junit.xml
        continue-on-error: true

      - name: Copy report out of container
        run: |
          CID=$(docker compose -f docker-compose.ci.yml ps -q backend)
          docker cp "$CID":/tmp/report.json ./report.json || true

      - name: Post results as PR comment
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          PASSED=$(jq '.summary.passed' report.json)
          FAILED=$(jq '.summary.failed' report.json)
          TOTAL=$(jq '.summary.total' report.json)

          if [ "$FAILED" -gt 0 ]; then
            STATUS="❌ **$FAILED failing** / $TOTAL total"
          else
            STATUS="✅ **All $TOTAL passing**"
          fi

          FAILING_LIST=$(jq -r '.tests[] | select(.outcome=="failed") | "- `\(.nodeid)`"' report.json)

          gh pr comment "${{ github.event.pull_request.number }}" --body "$(cat <<EOF
          ## Test Results — $STATUS

          | Passed | Failed | Total |
          |---|---|---|
          | $PASSED | $FAILED | $TOTAL |

          $([ "$FAILED" -gt 0 ] && echo "**Failing tests:**" && echo "$FAILING_LIST")

          <sub>Run: [\`${{ github.run_id }}\`](${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }})</sub>
          EOF
          )"

      - name: Report to test builder
        continue-on-error: true
        run: |
          curl -X POST "https://testbuilder.zubbystudio.shop/api/test-results" \
            -H "Content-Type: application/json" \
            -d @report.json \
            -d "{\"handoff_ref\": \"HO-027\", \"pr_number\": ${{ github.event.pull_request.number }}, \"commit_sha\": \"${{ github.sha }}\"}"

      - name: Fail job if tests failed
        run: |
          FAILED=$(jq '.summary.failed' report.json)
          [ "$FAILED" -eq 0 ]

  build:
    needs: tests
    runs-on: [self-hosted, epm]
    steps:
      - uses: actions/checkout@v4
      - name: Build backend image (no push — PR check only)
        run: docker build -t epm-backend:pr-${{ github.event.pull_request.number }} ./backend
      - name: Build frontend image (no push — PR check only)
        run: docker build -t epm-frontend:pr-${{ github.event.pull_request.number }} ./frontend
```

Note: the "Report to test builder" step has `continue-on-error: true` — the test builder endpoint may not exist yet. Do not let its failure block the PR check. Confirm in your HO response whether `testbuilder.zubbystudio.shop` currently resolves/responds; if not, either comment out the step or leave it failing-soft with a note.

## 5. Acceptance Criteria (for AT-*.md, to be verified before this is marked DONE)

1. Opening a PR against `test` triggers the workflow within the standard GitHub Actions latency (no manual dispatch required).
2. `static-analysis` job fails the check if `ruff` or `mypy` find violations.
3. `tests` job runs against `epm_test` inside the existing Postgres 15 instance — confirmed via `docker ps` showing no new Postgres container spun up during the run.
4. A PR comment appears with pass/fail counts and a list of failing test node IDs when applicable.
5. `build` job only runs if `tests` passes.
6. No step logs the raw value of `EPM_TEST_DB_PASSWORD`.
7. Re-running the workflow twice on the same PR does not leave stale/duplicate data in `epm_test` (truncation step is idempotent).

## 6. What NOT to do

- Do not add a `services:` block with any Postgres image.
- Do not use `actions/setup-python` — all Python execution stays inside the backend container image.
- Do not auto-merge or extend this workflow to touch `main`.
- Do not attempt to run or "test" this workflow locally on the Fedora workstation — validate via an actual draft PR on the VPS-connected runner instead.
