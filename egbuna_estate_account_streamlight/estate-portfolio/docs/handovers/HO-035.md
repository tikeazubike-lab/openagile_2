## 5. Key Technical Decisions Required (Claude to Approve)

### 5.1 ClaimRecord `claim_status` — New Values

The existing `claim_status` CHECK constraint uses: `pending`, `approved`, `rejected`, `partially_paid`, `paid`, `lapsed`.

The PO's 3-state lifecycle (`unresolved`, `unclaimed`, `claimed`) describes the **estate's recovery progress**, not the registrar's response status. This is a different dimension.

**Question**: Should we:
- **A)** Add a new `lifecycle_status` column (unresolved/unclaimed/claimed) alongside the existing `claim_status`, or
- **B)** Replace the `claim_status` CHECK constraint with the new 3-state set, or
- **C)** Use the existing `claim_status` with new values — map: unresolved→pending, unclaimed→approved, claimed→paid — and document the semantic shift?

**Recommendation**: Option A — additive change, preserves backward compatibility with any code already using `claim_status`. The new `lifecycle_status` column is what powers the claims dashboard filter and the registrar widget. The old `claim_status` continues to record registrar-level responses (approved by registrar, paid out).

### 5.2 Registrar Integration

The CSV `Operator` column contains registrar names (First Registrars, CardinalStone, etc.) that should be matched against the `Registrar` model's `name` field. No new `registrar_id` FK on ClaimRecord is needed — the chain is:

```
ClaimRecord → holding_id → Holding → company_id → Company → registrar_id → Registrar
```

**Question**: This chain already allows grouping claims by registrar on the registrars page, via a new endpoint (`GET /api/v1/registrars/{id}/claims` or aggregating in the claims list). Do you want a dedicated registrar-claims endpoint, or is an aggregated claims list with company→registrar projection sufficient for the widget?

### 5.3 Upload Endpoint Design

Following the cost_basis two-phase pattern:
- `POST /api/v1/claims/upload/preview` — parse CSV, match companies, return preview list with match confidence
- `POST /api/v1/claims/upload/commit` — commit the previewed records

**Question**: Agreed?

## 6. Implementation Plan (Pending Approval)

### Phase 1: Backend — Claims Upload Endpoint

1. **Model change**: Add `lifecycle_status` column (VARCHAR, CHECK unresolved/unclaimed/claimed) to ClaimRecord — nullable, default `'unresolved'`
2. **Fuzzy matcher**: Utility function `find_company(name: str) -> Company | None` using rapidfuzz against `Company.name`
3. **Upload router**: Add to `routers/claims.py`:
   - `POST /api/v1/claims/upload/preview` — parses CSV, fuzzy-matches companies, returns preview rows with match quality
   - `POST /api/v1/claims/upload/commit` — creates ClaimRecords with status=unresolved
4. **Template download**: `GET /api/v1/claims/upload/template` — returns CSV template

### Phase 2: Backend — Registrar Widget Data
