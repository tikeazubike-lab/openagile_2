# Bug Fix: PRIC-UPDATE-BE-E2E-001 — Stale Holding Values After PDF Price Upload

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Fix stale Holding.current_value and Holding.unrealized_gain_loss fields after PDF price upload, so dashboard and holdings pages show correct return % values.

**Architecture:** The bug is that `/api/v1/prices/upload-pdf` updates `Company.current_price` but never recalculates the denormalized `Holding.current_value` and `Holding.unrealized_gain_loss` fields. These fields are read directly by dashboard and holdings endpoints. Fix by adding a recalculation step in the PDF upload endpoint.

**Tech Stack:** FastAPI, SQLAlchemy async, PostgreSQL

---

## Bug Context

- **Bug ID:** PRIC-UPDATE-BE-E2E-001
- **Title:** Stale Holding Values After PDF Price Upload
- **Severity:** High (affects all price-dependent features)
- **Root Cause:** `Company.current_price` is updated during PDF upload, but `Holding.current_value` and `Holding.unrealized_gain_loss` are denormalized cached values that are never refreshed. Dashboard and holdings pages read these stale cached values directly.
- **Affected Endpoints:** `/api/v1/dashboard` (reads `h.current_value` directly), `/api/v1/holdings` (reads `return_pct` computed from `current_value`), holdings frontend

```python
# backend/app/services/holdings.py
from decimal import Decimal

def recalculate_holding_value(holding):
    """Recalculate holding's current_value and unrealized_gain_loss from company's current price."""
    if holding.company and holding.company.current_price is not None:
        price = Decimal(str(holding.company.current_price))
        shares = Decimal(str(holding.num_shares))
        cost = Decimal(str(holding.total_cost))
        
        holding.current_value = shares * price
        holding.unrealized_gain_loss = (shares * price) - cost
    else:
        holding.current_value = None
        holding.unrealized_gain_loss = None
```

## Risks / Tradeoffs

- **Performance:** Recalculating all holdings on every price upload adds DB overhead. Mitigation: only recalculate active holdings, batch in single query.
- **Data Consistency:** Denormalized fields (`current_value`, `unrealized_gain_loss`) are now kept in sync. Risk of drift is eliminated.
- **Historical Data:** If prices were uploaded before this fix, historical return_pct may still be wrong until next upload. Can run one-time migration.
